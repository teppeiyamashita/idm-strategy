# EUSP Engineering Team: Entry-Level Software Engineer

**Position Title:** Entry-Level Software Engineer (AI-Driven Implementation)  
**Reports To:** Principal Architect Engineering Manager  
**Team:** EUSP Engineering Team
**Experience Level:** 0-2 years professional software development  

---

## Executive Summary

The **Entry-Level Software Engineer** is a member of the EUSP Engineering Team responsible for translating detailed technical specifications into production-ready code using AI-assisted development tools. Under close mentorship from the Principal Architect, this role focuses on executing implementation work following established specifications, design patterns, and architectural guidance. This position is ideal for developers new to professional software development who are fluent with AI-assisted coding tools and committed to spec-driven, quality-focused engineering practices.

---

## Core Accountability

The engineer is accountable for:

1. **Detailed Design Development** – Working from Principal Architect's high-level design, develop detailed technical design: data models, API contracts, component architecture, integration patterns
2. **AI-Driven Implementation** – Translate detailed technical design and specifications into production code using AI-assisted development tools (GitHub Copilot, Claude Code)
3. **Spec Compliance** – Execute implementation strictly to detailed design and specification; request clarification when requirements are ambiguous
4. **Quality & Testing** – Write unit and integration tests derived directly from specification acceptance criteria
5. **Code Quality** – Follow established code standards, patterns, and conventions; actively participate in code reviews
6. **Learning & Growth** – Actively learn from Principal Architect guidance, code reviews, and team practices; master design and AI-assisted development workflows

## Spec-Driven Development Practices & Tools

**Core Principle:** All development activities execute specifications as written. Entry-level engineers implement to spec; they do not make design decisions.

**Mandatory Tool:** GitHub Spec Kit is required for all specifications—no exceptions.

### How Specs and Design Flow to Implementation
1. **Principal Architect Creates High-Level Design & Specification** using GitHub Spec Kit
   - Defines requirements, acceptance criteria (GHERKIN format)
   - Provides high-level design direction, technology choices, architectural pattern
   - Links to code examples and patterns to follow

2. **Entry-Level Engineer Develops Detailed Design**
   - Reviews high-level design and requirements
   - Asks clarifying questions on architectural direction
   - Proposes detailed design: data models, API contracts, component structure, integration approach
   - Documents detailed design in GitHub Spec Kit with rationale
   - Gets Principal Architect approval before implementing

3. **Entry-Level Engineer Implements to Detailed Design**
   - Uses GitHub Copilot/Claude Code as implementation assistant
   - Follows approved detailed design and architecture
   - Writes unit tests directly from specification acceptance criteria
   - Never changes design or specification scope without Principal Architect approval

4. **Principal Architect Reviews Design and Implementation**
   - Validates detailed design aligns with high-level direction and requirements
   - Validates code fulfills detailed design exactly
   - Ensures acceptance criteria are met through tests
   - Provides feedback and guidance
   - Approves only when design and implementation are complete and correct

### Key Principles for Entry-Level Engineers
- **High-Level Design is Direction:** Work from Principal Architect's high-level design; propose detailed design within that direction
- **Design Requires Approval:** Detailed design must be approved by Principal Architect before implementation begins
- **Implementation Matches Design:** Code must match approved detailed design exactly; if design and code differ, design wins
- **No Assumptions:** When in doubt, ask clarifying questions—never assume or infer requirements or design decisions
- **Design Improvements:** Suggest design improvements only with Principal Architect collaboration; never deviate without approval
- **Test from Criteria:** Tests are written directly from specification acceptance criteria (GHERKIN) before coding
- **Ask Before Deviating:** Any deviation from requirements, design, or specification requires Principal Architect approval

### Tools Entry-Level Engineers Must Know
- **Development Environment:** VS Code or Visual Studio with C#/ASP.NET Core extensions; comfortable with IDE debugger, intellisense, and integrated terminal
- **GitHub Spec Kit:** Reading specifications, understanding requirements and acceptance criteria, documenting detailed design decisions, and receiving/implementing feedback
- **Acceptance Criteria (GHERKIN):** Understanding Given-When-Then language in specs; writing clear acceptance criteria for detailed design
- **Test-Driven Development (TDD):** Write tests from spec acceptance criteria before or alongside code
- **GitHub Issues for Requirements Tracing:** Link code commits and pull requests to specification requirements and design decisions
- **Design Documentation:** Ability to document detailed design with diagrams, data models, API contracts, component structure

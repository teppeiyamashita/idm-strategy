# EUSP Engineering Team: Principal Architect

**Position Title:** Principal Architect (EUSP Engineering Team Lead)  
**Reports To:** Director of Engineering / VP Engineering  
**Team Leadership:** EUSP Engineering Team (1 Senior engineer + 1-2 entry-level engineers)  
**Experience Level:** 5+ years technical leadership experience  

---

## Executive Summary

The **Principal Architect** is the engineering leader of the EUSP Engineering Team responsible for technical vision, architectural design, team mentorship, and strategic decision-making. This role leads the design process end-to-end: understanding business requirements, creating high-level architectural designs, setting technical direction, and ensuring implementation quality. The Principal Architect partners with Principal Product Manager and business stakeholders to translate business requirements into robust, scalable solutions while building and mentoring a high-performing engineering team. This is a leadership role requiring deep technical expertise, architectural thinking, design authority, and strong mentoring capabilities.

---

## Core Accountability

The Principal Architect is accountable for:

1. **Design Leadership & Vision** – Own end-to-end solution design: translate business requirements into technical architecture, establish design patterns, set technical direction for the team
2. **High-Level Architectural Design** – Create detailed high-level designs with clear design decisions, rationale, and implementation guidance for engineers
3. **Specification & Requirements** – Create comprehensive specifications with acceptance criteria (GHERKIN) and clear implementation guidance using GitHub Spec Kit
4. **Team Development & Mentorship** – Lead Senior engineer on design decisions; close mentorship of entry-level engineers; establish engineering practices and standards
5. **Technical Excellence & Quality** – Establish code quality standards, testing practices, architecture review processes; ensure production quality
6. **Strategic Decision-Making** – Make technology choices, evaluate trade-offs, manage technical debt, communicate technical status to leadership

## Spec-Driven Development Leadership

**Core Principle:** All engineering activities are grounded in clear, written specifications. The Principal Architect is the architect of the specification-driven development process.

**Leadership Responsibility:** GitHub Spec Kit is non-negotiable; the Principal Architect enforces spec-driven development across the entire team.

### Specification Process Leadership
1. **Architect Collaborates with Product Manager** on requirements understanding
   - Clarify business objectives, user needs, constraints
   - Identify non-functional requirements: performance, scalability, security
   - Define scope and acceptance criteria with product

2. **Architect Creates High-Level Design & Specification** using GitHub Spec Kit
   - Document business requirements and acceptance criteria (GHERKIN)
   - Propose high-level architectural approach with rationale
   - Define technology choices and integration patterns
   - Provide code examples and patterns to follow
   - Document design decisions and trade-offs

3. **Architect Reviews Team Detailed Designs** (for Senior and entry-level engineers)
   - Validate detailed designs align with high-level direction
   - Ensure design soundness and adherence to architectural patterns
   - Approve detailed design before implementation begins
   - Provide feedback and guidance on design improvements

4. **Architect Reviews Team Implementations**
   - Validate code matches specification and approved detailed design
   - Ensure acceptance criteria are met through comprehensive tests
   - Provide feedback on code quality and architectural alignment
   - Approve pull requests only when implementation is complete and correct

### Design & Implementation Workflow Authority
- **Specification is Law:** All implementation must match specification exactly; specification changes require architect approval
- **Design Constraints:** High-level design decisions are not negotiable; engineers propose detailed design within constraints
- **No Vibe Coding:** Every feature, every component, every function begins with spec and design
- **Test-Driven from Spec:** Tests are written directly from acceptance criteria before or alongside code
- **Traceability Required:** Every requirement, design decision, test, and code segment is traceable to specification

### Tools & Framework Expertise
- **Development Environment:** VS Code or Visual Studio with C#/ASP.NET Core extensions; proficient with IDE debugging, code analysis, and integrated development workflows
- **GitHub Spec Kit:** Master-level proficiency; ability to train team on spec writing
- **Acceptance Criteria (GHERKIN):** Write and review clear Given-When-Then scenarios
- **Test-Driven Development (TDD):** Practice and preach TDD; ensure team practices TDD
- **Architecture Decision Records (ADRs):** Document significant technical decisions with context and rationale
- **Design Review Process:** Facilitate architecture discussions; lead design validation
- **GitHub Issues for Requirements Tracing:** Establish traceability patterns for team

## Executive Summary

## Spec-Driven Development Practices & Tools

**Core Principle:** All development activities execute specifications as written. Mid-career engineers propose detailed designs within high-level architectural constraints set by the Principal Architect.

**Mandatory Tool:** GitHub Spec Kit is required for all specifications—no exceptions.

### How Specs and Design Flow to Implementation
1. **Principal Architect Creates High-Level Design & Specification** using GitHub Spec Kit
   - Defines requirements, acceptance criteria (GHERKIN format)
   - Provides high-level design direction, technology choices, architectural pattern
   - Links to code examples and patterns to follow

2. **Senior Engineer Proposes Detailed Design**
   - Reviews high-level design and requirements
   - Asks clarifying questions on architectural direction
   - Proposes detailed design: data models, API contracts, component structure, integration approach
   - Documents detailed design in GitHub Spec Kit with rationale
   - Gets Principal Architect approval before implementing

3. **Senior Engineer Implements to Detailed Design**
   - Uses GitHub Copilot/Claude Code as implementation assistant
   - Follows approved detailed design and architecture
   - Writes unit tests directly from specification acceptance criteria
   - Never changes design or specification scope without Principal Architect approval

4. **Principal Architect Reviews Design and Implementation**
   - Validates detailed design aligns with high-level direction and requirements
   - Validates code fulfills detailed design exactly
   - Ensures acceptance criteria are met through tests
   - Provides feedback and guidance
   - Approves only when design and implementation are complete and correct

### Key Principles for Senior Engineers
- **High-Level Design is Constraint:** Work from Principal Architect's high-level design; all detailed designs must honor architectural direction
- **Design Proposals:** Propose detailed design alternatives within high-level direction; obtain approval before implementing
- **Implementation Matches Design:** Code must match approved detailed design exactly; if design and code differ, design wins
- **No Assumptions:** When in doubt, ask clarifying questions—never assume or infer requirements or design decisions
- **Design Improvements:** Suggest design improvements only with Principal Architect collaboration; never deviate without approval
- **Test from Criteria:** Tests are written directly from specification acceptance criteria (GHERKIN) before or alongside coding
- **Ask Before Deviating:** Any deviation from requirements, design, or specification requires Principal Architect approval

### Tools Senior Engineers Must Know
- **Development Environment:** VS Code or Visual Studio with C#/ASP.NET Core extensions; comfortable with IDE debugger, intellisense, and integrated terminal
- **GitHub Spec Kit:** Creating detailed design proposals, understanding requirements and acceptance criteria, writing specifications, and providing design feedback
- **Acceptance Criteria (GHERKIN):** Understanding Given-When-Then language in specs; writing clear acceptance criteria for detailed design
- **Test-Driven Development (TDD):** Write tests from spec acceptance criteria before or alongside code
- **GitHub Issues for Requirements Tracing:** Link code commits and pull requests to specification requirements and design decisions
- **Design Documentation:** Ability to document detailed design with diagrams, data models, API contracts, component structure
